# 설계

## backend

- Restful api를 제공
- base_path : /pcms
- jwt로 auth
- 중복로그인 배제
- redis로 로그아웃시 즉시 로그아웃기능 구현
- 아래와 같은 properties를 사용한다.

```text
# folders
pcms.base-dir=./pcms_data
file.upload.base-dir=${pcms.base-dir}/uploads
file.upload.attach-files-dir=${file.upload.base-dir}/attachments
file.upload.editor-images-dir=${file.upload.base-dir}/editor-images
file.temp-dir=${pcms.base-dir}/temp
# image base url
file.image.base-url=http://localhost/pcms/files/images/
# apnode base dir
apnode.file.base-dir=${pcms.base-dir}/apnodes
apnode.file.base-url=http://localhost/pcms/apnodes/
```

## api 설계

- 주소창에서의 주소는 http://.../pcms/auth/login 과 같은 상황임

### auth

- 로그인 : post /auth/login
- 로그아웃 : post /auth/logout
- Refresh Token : get /auth/refesh
- 패스워드 암호화는 개발하는 동안에는 평문으로 비교

### diary

- table `diary` 에 crud를 구현한다.
- Create : post /diary
- Edit(수정) : put /diary
- Delete   : delete /diary/{id}
- View     : get /diary/{id}
- List     : get /diary
  - param으로 페이징관련된 값들을 받는다.
  - 날짜 범위를 start_ymd와 end_ymd를 검색조건으로 받는다.
  - keyword를 검색조건으로 받는다.

- 편집기 내용 및 첨부파일의 저장
  - TipTap을 이용한 html편집기 사용
  - 저장시또는 수정시 내용에 포함된 이미지를 정해진 application-<profile>.properties에 포함된 file.upload.editor-images-dir에 저장하고 url로 치환해서 content를 만들어서 저장한다.
  - file.upload.attach-files-dir에  yyyy/mm/dd의 폴더를 만들어서 그 아래 첨부파일을 저장한다.
  - 첨부파일은 files와 file_match 테이블에 의해서 관리된다.

### 파일관리

- 웹 UI 로 파일시스템을 구현한다.
- 유닉스 파일시스템을 모방하여 테이블을 설계함.
- 2개의 테이블로 설계 : ap_node, ap_file를 사용한다.
- ap_node는 self-relation을 갖는다.
- ap_file은 저장된 파일에 대한 정보를 갖는다.
- 아래와 같은 정보를 바탕으로 파일의 저장위치를 갖는다.
  - yyyy/mm/dd의 폴더를 만들어서 그 아래 첨부파일을 저장하는 것은 동일하다.

```text
apnode.file.base-dir=${kalpa.base-dir}/apnodes
apnode.file.base-url=http://jskn.iptime.org/apnodes/
```

### 파일관리 API

> 테이블: `ap_node` (트리 노드) + `ap_file` (파일 메타데이터, 1:1)
> 컨텍스트: `/apnode` — 인증 필요 (authenticated)

#### 1. 노드(트리) 탐색

| Method | URL | 설명 |
|--------|-----|------|
| `GET` | `/apnode` | 루트 디렉토리 목록 (parent_id IS NULL) |
| `GET` | `/apnode/{id}/children` | 특정 디렉토리의 자식 노드 목록 |
| `GET` | `/apnode/{id}` | 노드 단건 조회 (파일이면 ap_file 정보 포함) |
| `GET` | `/apnode/{id}/breadcrumb` | 현재 노드 → 루트까지 경로 (네비게이션) |

#### 2. 디렉토리 관리

| Method | URL | 설명 | Body |
|--------|-----|------|------|
| `POST` | `/apnode/directory` | 디렉토리 생성 | `{ parentId, name }` |
| `PUT` | `/apnode/{id}/rename` | 노드 이름 변경 | `{ name }` |
| `PUT` | `/apnode/{id}/move` | 노드 이동 (depth ≤ 20 검증) | `{ newParentId }` |
| `DELETE` | `/apnode/{id}` | 노드 소프트 삭제 (is_deleted=true) | — |
| `PUT` | `/apnode/{id}/restore` | 삭제된 노드 복구 | — |

#### 3. 파일 업로드/다운로드

| Method | URL | 설명 |
|--------|-----|------|
| `POST` | `/apnode/{parentId}/files` | 파일 업로드 (multipart, 다중) → ap_node(F) + ap_file 생성 |
| `GET` | `/apnode/{id}/download` | 파일 다운로드 (Content-Disposition) |
| `GET` | `/apnode/{id}/thumbnail` | 이미지 썸네일 조회 |

#### 4. 링크

| Method | URL | 설명 | Body |
|--------|-----|------|------|
| `POST` | `/apnode/link` | 링크 생성 | `{ parentId, name, linkTargetId }` |

#### 5. 유틸리티 (선택)

| Method | URL | 설명 |
|--------|-----|------|
| `GET` | `/apnode/{id}/tree` | 하위 트리 전체 조회 (재귀, 프론트 트리뷰용) |
| `GET` | `/apnode/search?keyword=xxx` | 이름 검색 |
| `DELETE` | `/apnode/trash` | 휴지통 비우기 (is_deleted=true 물리 삭제) |
| `GET` | `/apnode/trash` | 삭제된 노드 목록 (휴지통) |

#### 설계 메모

- 소프트 삭제: DELETE 시 `is_deleted=true` + `delete_dt` 설정, 실제 파일은 휴지통 비우기에서 삭제
- depth 검증: 이동/생성 시 depth ≤ 20 제약조건을 서비스 레이어에서 사전 검증
- SHA256: 업로드 시 해시 계산, 중복 파일 감지에 활용 가능
- child_count / total_size: 디렉토리 캐시 필드 — 파일 추가/삭제 시 서비스에서 갱신
- 구현 순서: 탐색 → 디렉토리 관리 → 파일 업로드/다운로드
